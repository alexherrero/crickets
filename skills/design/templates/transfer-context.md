## What you (Gemini in Antigravity) should do

1. **Wait** for the operator to add inline comments to the target doc using Antigravity's native inline-comment UI.
2. When the operator asks you to apply (typical phrasing: "apply my comments" / "revise per my notes" / "implement my feedback"), proceed:
   - Read each inline comment in document order.
   - Apply the requested change while preserving the doc's existing structure + conventions per the sections above.
   - Save the revised doc to the same path.
3. Write a **change-summary log** to `<TARGET_DOC>.diff.md` (a sibling file to the target doc) listing:
   - One section per applied comment: location (line / section) + the operator's comment text + your applied change + any deviation from the comment's literal ask (with rationale).
   - A "Comments not applied" section if any comment was rejected or partially applied (with rationale).
   - A "Suggestions" section if you noticed adjacent issues the operator may want to address in a future pass.

## What you MUST NOT do

- **Change frontmatter** (`status`, `visibility`, `created`, `last_major_revision`, etc.) **unless an inline comment explicitly says to**.
- **Add new top-level sections** the operator didn't request via inline comment.
- **Remove existing sections** unless an inline comment explicitly says to.
- **Modify pre-handoff Document History rows** — those are immutable audit trail. You can ADD a new row at the bottom noting the external-review pass; you cannot rewrite past rows.
- **Revert any "Recent decisions to honor" above** without an explicit override comment from the operator (and if you do, flag the override prominently in the change-summary log).
- **Apply silent improvements** beyond the inline comments. If you spot something worth fixing that wasn't commented on, put it in the change-summary log's "Suggestions" section — never apply it without an operator comment.

## When done

The operator will return to Claude Code and say one of:

- `"review complete on <slug>"` (natural language)
- `/design author <slug> --resume-external-review` (explicit slash invocation)
- `/plan --resume-external-review` (for PLAN.md handoffs)

Claude Code will:

1. Read the revised target doc + the `<TARGET_DOC>.diff.md` change-summary log.
2. Diff against the pre-handoff snapshot at `<SNAPSHOT_PATH>`.
3. Surface findings to the operator: what changed, any unexpected modifications, anything in the change-summary "Suggestions" worth promoting to an additional revision pass.
4. Ask the operator whether to accept all changes / iterate further / discard the handoff entirely.
5. If accepted: clean up the snapshot + transfer-context files; continue the original flow (e.g. proceed to Status transition for `/design author`; proceed to `/design translate` for finalized design docs; proceed to `/work` for refined PLAN.md).
6. If iterate: regenerate transfer-context + offer another handoff.

## How to resume

After this handoff is complete + the cycle returns to Claude, the cycle can repeat. Each subsequent handoff regenerates a fresh transfer-context file with updated "Recent decisions to honor" (including whatever got applied in the previous round). This prevents stale-context drift across multi-round iterations.
