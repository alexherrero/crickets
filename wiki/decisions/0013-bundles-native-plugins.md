# ADR 0013 — Bundles are native host plugins, generated from one source of truth

> [!NOTE]
> Status: accepted
> Date: 2026-06-02

## Context

crickets ships the operator's developer-workflow customizations — skills, sub-agents, hooks, slash commands, MCP configs — into Claude Code and Antigravity. Through v2.x it did this with a bespoke `install.sh` that parsed each primitive's YAML frontmatter and copied it to host-specific paths, and it shared its `lib/install/` layer byte-for-byte with the sibling `agentm` repo. Both target hosts have since shipped *native* plugin systems that already do dispatch, dependency resolution, and distribution. Crickets v3.0 (#40) reframes a "bundle" as a **native host plugin generated from a single source of truth**. The companion HLD [crickets-v3-native-plugins](../designs/crickets-v3-native-plugins) ships the full design; this ADR locks the load-bearing decisions.

**Open questions the decision resolves:**

- How do we ship grouped customizations to two hosts without hand-maintaining two copies that drift?
- Where does a "bundle" live — a hand-authored per-host artifact, or a generated one?
- Is generated output committed, or produced at install time?
- How is cross-plugin reuse (Testing builds on Developer) expressed without duplicating primitives?

## Decision

### 1. One source of truth plus a generator that emits native per-host plugins

Authors write each customization once under `src/<group>/` (folder-per-group, primitive frontmatter carrying `kind` / `supported_hosts` / `requires`). A deterministic, stdlib-only `scripts/generate.py` reads `src/` and emits, per functional group, a native plugin directory **per host** — `dist/claude-code/plugins/<group>/` and `dist/antigravity/plugins/<group>/` — plus each host's marketplace manifest.

**Why not hand-author per-host plugins?** That recreates the adapter-tree duplication [ADR 0006](0006-gemini-cli-host-removal) split *away from* — two hand-maintained copies that drift. The generator is the single-source discipline.

**Why not generate at install time (ship only the SoT)?** The marketplace must serve static files, and committed artifacts give an auditable CI diff plus let users browse the real plugins before installing.

### 2. Functional grouping is the unit of distribution ("plugin")

Primitives are organized into functional groups — Developer (the base), GitHub-CI, PII, Wiki (and, in bucket ④, Testing / Releasing / knowledge). A primitive's group is simply the folder it lives in. Cross-plugin reuse is expressed by **plugin dependencies at generation time**, never by a primitive belonging to two groups, so no file is duplicated.

**Why group-as-folder?** The grouping is legible straight from the tree, and group-level metadata (`group.yaml`: name / description / category / `requires` / `standalone`) is cleanly separated from per-primitive frontmatter.

### 3. Committed `dist/` with a generated-in-sync CI gate

The generated artifacts are committed. `generate.py check` re-runs the generator into a temp dir and fails if the committed output differs — the drift gate that replaces `check-lib-parity.sh`. Determinism (sorted iteration, stable JSON, no timestamps) is therefore mandatory, not cosmetic.

**Why not trust authors to rebuild?** A human-forgettable rebuild step is exactly how generated output silently drifts from source. The gate makes drift a red CI, not a latent bug.

### 4. Antigravity composition is thin-separate; Claude uses native dependencies

A group's `requires:` emits differently per host. Claude gets native `dependencies` (installing a dependent auto-installs its base). Antigravity carries only the dependent's own primitives, with the requirement documented in the marketplace entry ("install Developer first").

**Why not native deps on Antigravity too?** At design time Antigravity's cross-plugin dependency support was unconfirmed. **Re-audit closed at the #40 dogfood:** `agy` 1.0.2's plugin.json carries no `dependencies`/`requires` field and the CLI exposes no dependency resolution — thin-separate is correct; no generator change.

## Consequences

### Positive

- **Single-source discipline.** One edit in `src/`, regenerate, both hosts stay in lockstep — no per-host drift.
- **Native machinery for free.** Dispatch, dependency resolution, and distribution come from the hosts' own plugin systems; crickets stops re-implementing them.
- **Auditable releases.** Committed `dist/` + the `check` gate make every generated byte reviewable in a PR diff.
- **Extensible to new hosts.** Adding a host = adding an emitter module behind the common `HostEmitter` interface.

### Negative

- **The generator is a maintenance surface.** Per-host emitters encode each host's manifest/hook/marketplace shape; a host schema change means an emitter change. Mitigated by the dogfood + the `check` gate.
- **Determinism is load-bearing.** Any nondeterministic emitter output (timestamps, set ordering) breaks the `check` gate. Enforced by convention + tests.
- **A repo-root marketplace pointer lives outside `dist/`** (for the one-word GitHub install, part-6 T3). It's generator-emitted and the `check` gate covers it explicitly so it can't drift.

### Load-bearing assumptions + re-audit triggers

1. **Both hosts keep native plugin systems with compatible-enough shapes.** Re-audit if Claude Code or Antigravity changes its plugin manifest or marketplace schema (the emitters + committed `dist/` would need a regenerate + dogfood).
2. **Antigravity plugin hooks are observe/side-effect-only.** Dogfood finding (#40 T2): `agy` 1.0.2 ignores a hook's exit code *and never reads its stdout*, so veto/inject hooks (`kill-switch`, `steer`) cannot enforce on Antigravity — they are Claude-only-effective; side-effect hooks (`commit-on-stop`) work on both. We ship all three + document the limitation (see [Compatibility](../reference/Compatibility)). **Re-audit if Antigravity ships hook-veto support** (a stdout decision schema or honored exit code) — then `kill-switch`/`steer` can enforce on Antigravity and the limitation note retires.
3. **Determinism holds.** Re-audit if any host ever requires nondeterministic plugin output (the `check` gate would need a different invariant).
