# ADR 0018 — Per-folder sidebars: intent-grouped wiki folders with collapse/expand nav

> [!NOTE]
> **Status:** accepted · **Date:** 2026-06-08

## Context

The wiki shipped with a single `_Sidebar.md` that GitHub renders on every page — the complete sitemap, grouped by intent (Get started · Do · Reference · Why it works · Designs · Decisions). As the wiki grew that one sidebar got long, and it couldn't focus: every page showed the whole tree, with no sense of which section you were in. We wanted **contextual navigation** — collapse the sections you're not in, expand the one you are — without leaving GitHub Wiki for a heavier docs platform.

Open questions this resolves:

- Can GitHub Wiki show a page-specific sidebar at all, or is it always the one root `_Sidebar.md`?
- If we organize by section, what *is* a "section" — Diátaxis type (`reference/`, `how-to/`) or user intent (Get started, Do)?
- How do we keep Diátaxis mode-enforcement when an intent folder mixes modes?
- Does every page still need to be listed on the root sidebar?

## Decision

**Organize the wiki into intent-group folders — one per nav section — each carrying its own `_Sidebar.md`.** GitHub Wiki renders the **nearest** `_Sidebar.md` walking up the directory tree (verified empirically: a page under `reference/` is served `reference/_Sidebar.md`; a root page gets the root one). Each folder's sidebar shows the full section list with **only that section expanded**; collapsed sections link to a representative entry page. The folders are `get-started/ · do/ · reference/ · why/ · designs/ · decisions/`.

- **Intent folders, not Diátaxis-type folders.** The nav is intent-grouped and that ordering is deliberate; type folders would force a coarser type-based nav and split intent groups (Get started spans a tutorial *and* a how-to). With intent folders the folder *is* the nav section, so the per-folder sidebar is the section's nav. *Not* type folders — they'd make the sidebar contradict the intent ordering.
- **Per-folder `_Sidebar.md`, not a dynamic sidebar.** GitHub Wiki has no client-side nav; one `_Sidebar.md` cannot be page-aware. Per-folder sidebars are the only way to get contextual nav on GitHub Wiki. *Not* a JS/Docusaurus-style sidebar — that's a platform migration we explicitly avoided.
- **A `<!-- mode: X -->` hint, not folder-derived mode.** `check-wiki` inferred a page's Diátaxis mode from its folder; intent folders mix modes (a tutorial in `get-started/`, how-tos under `reference/`). Rather than drop mode-enforcement, a page whose folder default doesn't fit carries an invisible mode hint the linter reads. *Not* a content heuristic (fragile) and *not* relaxing the mode-shape rules (loses the single-mode discipline).
- **≤2-levels reachability, not "sidebar = complete sitemap".** The root lists the sections; each folder lists its own pages — a page is reachable in two clicks without being on the root. `check-wiki` rule-j now checks the **union** of all `_Sidebar.md` (no orphans) instead of root-lists-everything.

## Consequences

**Positive:**

- Contextual collapse/expand nav on plain GitHub Wiki — no platform change, no JavaScript.
- Basename links are move-safe, so the ~70-file reorg needed **zero** wiki-internal link changes.
- The folder layout mirrors the nav, so the structure is legible straight from the tree.
- Composes cleanly with the wave-2 per-plugin model: a plugin folder + its `plugin-home` index nests under a section.

**Negative:**

- N hand-maintained `_Sidebar.md` files that must stay in sync — to be **generated by the composer** (wave-2), not hand-authored long-term.
- The `wiki-sync` dupe-check had to exempt `_Sidebar.md`/`_Footer.md` (they legitimately share a basename).
- Cross-mode pages carry a `<!-- mode: -->` hint — a small authoring tax (3 pages today).
- Antigravity's preview doesn't render basename links as clickable and its per-directory sidebar behaviour is unverified — the benefit is GitHub-Wiki-specific.

**Load-bearing assumptions (re-audit triggers):**

- **GitHub Wiki renders the nearest `_Sidebar.md`** for a page by its directory. An under-documented Gollum feature, verified empirically 2026-06-08 (`do/`→Do, `why/`→Why it works, `decisions/`→Decisions all expanded correctly). **Re-audit if** GitHub changes wiki rendering, migrates off Gollum, or the per-folder sidebars stop taking effect — fall back to a single root sidebar, or move to a docs platform with native nav.
- **`rsync -a` in `wiki-sync.yml` preserves the directory tree** into the GitHub wiki repo (the prerequisite for nearest-`_Sidebar`). **Re-audit if** the sync ever flattens the tree.
