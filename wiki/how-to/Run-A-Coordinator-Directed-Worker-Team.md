# How to run a coordinator-directed worker team

> [!NOTE]
> **Goal:** Run the full coordinator loop as one operator â€” take a brief through research, plan-authoring, isolated execution, integration, and a queue read â€” using the role roster as personas over the phase commands you already have. This is the capstone that ties the four V5-10 worker-team siblings together.
> **Prereqs:** the `developer-workflows` plugin installed ([Install crickets plugins](Install-Into-Project)) at 0.8.0 or later (the version that ships the role roster); a clean `main` working tree; a brief â€” the thing you want built. Read [Coordinator roles](Coordinator-Roles) first for what each role wraps and which are read-only.

You are the **coordinator**. The roster gives you four personas â€” `researcher`, `tech-lead`, `worker`, and `project-manager` â€” that are thin skins over the shipped phase commands ([Coordinator roles](Coordinator-Roles)), and this page walks through them once, in order, cross-linking each step's own recipe. Each role wraps a command you can also run bare; the roster just names the persona doing it.

## Steps

1. **Research the brief (`researcher`).** Front the loop with the read-only `researcher` role to scope the work before any plan exists. It wraps `explorer` for an in-repo fan-out ("where does X live / what tests cover Z") and runs its own `WebFetch` for a few targeted web lookups (a spec / API / changelog check). For deeper multi-source research it forward-references the operator's global research agent (e.g. a personally-installed `memory-idea-researcher`). It composes with that agent when one is present, and never vendors or ports it. When none is installed, it degrades gracefully to `explorer` plus its own `WebFetch`. Adopt the `researcher` persona (or paste its def from `src/developer-workflows/agents/researcher.md`) and ask your scoping question; it returns a structured finding â€” a 1â€“3 sentence answer, the `file:line` / web sources backing it, and the open questions a planner needs. The output is a scoped brief, not a plan; because the role is read-only it makes no edits.

2. **Author the plan (`tech-lead`).** Hand the scoped brief to the active `tech-lead` role â€” the brief-to-plan authoring persona (full tool access). For a brief that warrants the fuller design-doc treatment, run `/design` first (author â†’ translate â†’ sequence, shipped); otherwise its floor is `/plan` directly. Author a named plan and stage it into the inactive tier, then promote it when you are ready to run it: `/plan --name <slug>` (or `--stage <slug>` to park it in `queued-plans/`, invisible to `/work` and `/queue-status-lite`), then `/plan --activate <slug>` to promote the staged plan to the active `PLAN-<slug>.md` (no-clobber guarded). Staging then activating lets you queue several plans and turn them on one at a time without singleton collisions. See [Run a named plan](Run-A-Named-Plan) for the full staging / activation surface.

3. **Run `/work` â€” it spawns the worker's worktree for you.** With `isolation.mode: worktree-per-plan` set in `.harness/project.json` (or an explicit operator instruction), running `/work` against the activated named plan auto-spawns the worktree at step 1.5, using the host's own worktree primitive â€” Claude Code's `EnterWorktree`, or Antigravity's New-Worktree-Mode / `invoke_subagent`. The worktree lands at `.claude/worktrees/<name>` on a branch named `worktree-<name>`, and `worktree_marker.py` writes the worktree-local `.harness/active-plan` marker that binds it to the plan. There's no separate spawn command â€” worktrees are still authority-gated (an explicit operator instruction or the durable config opt-in, never autonomous), just via `/work` itself rather than a dedicated command. Full recipe: [Run a named plan](Run-A-Named-Plan).

4. **Execute the plan (`worker`).** The active `worker` role is the persona of this `/work` session. One worker runs per worktree, works its plan's **full** task list autonomously, and stays single-threaded. Each task is gated by the per-task safety pre-check â€” it works straight through and only stops to ask on a hard-to-reverse, ambiguous, scope-drifting, or unverifiable task. It gates green before every `[x]`, commits one task per commit, and updates `PLAN-<name>.md` + `progress-<name>.md` as it goes. It never fans out parallel implementers â€” single-threaded execution is the safety constraint the whole loop rests on. Several workers can run concurrently, each in its own worktree bound to its own plan, without colliding.

5. **`/work` closes the loop itself, on the plan's final task.** `finalize_unit.py` pushes the worktree's branch, opens a PR carrying the plan's close-out summary as its description, and arms `gh pr merge --auto --squash`. The PR merges once its required checks â€” the full deterministic battery, gated by branch protection â€” go green, with no further operator invocation; a red check simply blocks the merge, so there's nothing to roll back by hand. `/work` then runs `ExitWorktree keep`, since the branch still has an open PR against it. A periodic sidecar, `worktree_shepherd.py`, reclaims worktrees left behind after their PR merges and rebases any PR GitHub reports as `BEHIND` its base.

6. **Read the queue (`project-manager`).** At any point, adopt the read-only `project-manager` role to look over the whole queue. It wraps `/queue-status-lite`, which lists every active plan â€” its binding, its latest progress line, and the state of any worker worktree, read through the `doctor_worktrees.py` probe. The role only shows you that render; it changes nothing on disk. It is a glance, not a gate: merge order stays a human decision (LC-5), so what to `/work` next is your call. **crickets #41** (github-projects board-sync â€” a synced board view) has landed and composes with this glance; **V5-11** (the chief-of-staff intelligence layer â€” `/standup`, readiness signals, integration-order *advisory*, still advisory) remains forward-referenced. Full recipe: [See every active plan](See-Every-Active-Plan).

## Verify

- Each step's verify is the verify of the command it drives â€” see the per-step how-tos linked above.
- After step 2, `/plan --activate <slug>` leaves an active `PLAN-<slug>.md`; after step 3, the `worktree-<name>` worktree exists with a `.harness/active-plan` marker holding the bare slug.
- The whole loop is complete when the plan's PR merges (the full gate battery passing as the required check on the merge), and a fresh `project-manager` glance via `/queue-status-lite` shows the plan dropped from the active queue.

## Troubleshooting

- **`/work` can't find a plan inside the worktree.** The `.harness/active-plan` marker binds it â€” confirm you're inside the worktree `/work`'s auto-spawn step created, not back on `main`. Run `/work` bare there; do not pass `--name`.
- **The plan's PR won't merge.** The gate battery is a required status check on the PR â€” a red run simply blocks the merge until it's fixed. Fix the failure in the worker's worktree, push again, and let the check re-run; there's no separate rollback step, since nothing merged in the first place.
- **`project-manager` shows a stale or orphaned plan.** The PM only reads; it never prunes. Use the `doctor_worktrees.py` probe / [See every active plan](See-Every-Active-Plan) to spot an orphaned worktree â€” `worktree_shepherd.py` reclaims it automatically once it's old enough and provably safe, or clean it up by hand through the worker lifecycle. The PM will not do it for you.
- Per-command failure modes (spawn guards, PR-merge gating) live on each step's how-to; this page only sequences them.
