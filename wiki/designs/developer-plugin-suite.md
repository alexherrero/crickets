# Developer Plugin Suite — workflows · safety · code-review + the `enhances:` composition schema

## Context

### Objective

crickets v3.x **bucket ④ (Wave 1, critical)** extracts the operator's opinionated developer process out of agentm into native host plugins — the first build step of the **V5 "unbundling"** ([memory-os-architecture HLD](https://github.com/alexherrero/agentm/wiki/memory-os-architecture)). This design specifies the **critical-3** dev-loop plugins plus the schema change that lets them compose: **`developer-workflows`** (the phase-gated `research→design→plan→work→review→release→bugfix` loop), **`developer-safety`** (control/safety hooks usable in *any* session), and **`code-review`** (adversarial review of any diff/PR). They share **no hard dependency** — each is usable alone — but compose into a richer experience when installed together, via a **new soft-composition manifest field, `enhances:`**. It's ready to build now because #40 shipped the generator and Hardening I made agentm a clean standalone; the plugins are built **parallel-run against today's agentm** (they coexist with the baked-in copies until the ⑤ V5 slim).

### Background

- **Where this sits.** Bucket ④ of ROADMAP-MASTER's 12-bucket re-sequence (`② Hardening I → ④ critical-3 → ⑤ V5 split → ⑥ rest`). It's the dev-loop foundation the operator **dogfoods to run the rest of the program** — including the ⑤ V5 kernel split itself. Per the [memory-os-architecture HLD](https://github.com/alexherrero/agentm/wiki/memory-os-architecture), agentm becomes a storage-agnostic memory OS + plugin host; the engineering process unbundles into crickets plugins, and these three are its base layer.
- **#40 gave us the generator.** crickets v3.0 ([crickets-v3-native-plugins.md](crickets-v3-native-plugins.md)) ships `src/<group>/` → `scripts/generate.py` → committed `dist/<host>/` native plugins + the composition vocabulary (`requires:` / `standalone:`). The seed `developer` group (3 control hooks + the `evaluator` agent) is the starting point — this design **splits and fleshes it**.
- **What's being extracted** lives in agentm today: the 5 phase specs + `bugfix` pipeline + `principles.md`, the phase commands, the dev sub-agents (adversarial-reviewer ×2, explorer, evaluator), the dev hooks (evidence-tracker, harness-context), the `.harness/` scaffold templates, and the dev conventions. The full boundary map is in the HLD + `ROADMAP-AgentMemoryV5.md`.
- **The `enhances:` gap.** `requires:` / `standalone:` model only *hard* dependencies. The relationships here are *soft* — "auto-engage when present" (safety → workflows; code-review → workflows' `/review`). That had no manifest expression; this design adds `enhances:` as the general soft-composition mechanism for the whole catalog.
- **Carried #40 constraints (load-bearing).** Antigravity has **no native cross-plugin deps** (so `requires:`/`enhances:` are *tooling*-resolved, not host-resolved); **AG plugin hooks are observe/side-effect-only** (kill-switch/steer are Claude-only-effective; commit-on-stop works on both).

## Design

### Overview

Three native host plugins, generated by the #40 chain from `src/<group>/` into `dist/<host>/`, each **`standalone: true`**:

- **`developer-workflows`** — the phase-gated loop: the 5 phase specs + `bugfix` pipeline + `principles.md`; the `/setup /plan /work /review /release /bugfix` commands; the `explorer` + `evaluator` sub-agents; the `harness-context` hook; the `.harness/` scaffold templates; and the workflow conventions. Its **`/review` is thin** — it orchestrates (gather diff, run deterministic gates) and dispatches a deeper adversarial pass *only if `code-review` is present*.
- **`developer-safety`** — the universal control layer: the `kill-switch`, `steer`, `commit-on-stop` hooks + the `commit-no-coauthor` / `worktrees-never-auto` conventions. Usable in *any* session, workflow or not.
- **`code-review`** — adversarial code review: the `adversarial-reviewer` + `adversarial-reviewer-cross` agents, the `evidence-tracker` hook, `cross-review.sh`, and a standalone **`/code-review <diff|PR>`** command. Reviews any diff with no `/work` cycle.

None `requires:` another. They **compose via a new manifest field, `enhances:`** — `developer-safety` enhances `developer-workflows` (its hooks engage across the loop); `code-review` enhances `developer-workflows`' `review` capability (the thin phase gains the adversarial pass). Install one, two, or all three — each adds capability without breaking the others. The `developer` seed splits to seed these (control hooks → safety, `evaluator` → workflows); the existing `wiki` / `github-ci` groups re-point their `requires:` from `developer` to `developer-workflows`.

### Infrastructure

No new runtime infrastructure — the #40 generator chain carries it. The only net-new *build* surface is the `enhances:` field in the `group.yaml` schema + its handling across `src_model.py` / `lint_src.py` / the emitters / `bootstrap.sh`.

```
src/developer-workflows/  ┐
src/developer-safety/     ├─ scripts/generate.py ─► dist/<host>/<plugin>/   (native, drift-gated by `generate.py check`)
src/code-review/          ┘                          + claude-code & antigravity marketplaces
   group.yaml: standalone: true · requires: [] · enhances: [ … ]
```

Built **parallel-run**: the plugins land while agentm still ships the baked-in copies; the ⑤ V5 slim removes the duplicates after dogfood-proof. **No host-level dependency wiring** — Claude's native hard-`dependencies` are unused (all three are standalone) and Antigravity has none; composition is *our* `enhances:` concept, **tooling-resolved**, not host-resolved.

### Detailed Design

#### 1. The `enhances:` composition schema (+ generator/lint)

A new optional `group.yaml` field expressing **soft, progressive-enhancement** composition — declared on the **enhancer** (keeps the enhancee open/extensible: new plugins can enhance it without editing its manifest):

```yaml
# code-review/group.yaml
name: Code Review
standalone: true
requires: []
enhances:
  - group: developer-workflows
    capability: review            # optional — targets a specific capability
    effect: "/review dispatches adversarial-reviewer + adversarial-reviewer-cross"
```

- **Orthogonal to `requires:`/`standalone:`.** The `standalone: true ⟺ requires: []` invariant stays (hard deps); `enhances:` is the soft axis, allowed on standalone plugins.
- **Capability targeting.** The enhancee optionally declares `capabilities: [setup, plan, work, review, release, bugfix]` in its `group.yaml`; an `enhances` entry's `capability:` validates against that list.
- **`lint_src.py` rules:** target group must exist · no self-enhance · `enhances ∩ requires = ∅` · named `capability` must be a declared capability of the target.
- **Consumers:** `bootstrap.sh` (suggests installing an enhancer — "you have developer-workflows; install developer-safety / code-review for the full experience"), the marketplace render ("works better with …"), and the docs. The field is **declarative metadata**; the actual enabling is §5.

#### 2. `developer-workflows` (the phase-gated loop)

Extracts from agentm: `harness/phases/{01-setup…05-release}.md` + `pipelines/bugfix.md` + `principles.md`; the per-host phase commands; the `explorer` + `evaluator` sub-agents; the `harness-context-session-start` hook; the `.harness/` scaffold (PLAN/features/progress/init/verify), written at runtime by `/setup`. Carries the workflow conventions (`phases-one-per-session`, `work-single-task`, `work-loop-grouping`, `pre-approval-batch`, `subagents-read-only`, plan-md/status-report shapes, `wake-on-ci`). `standalone: true`; declares its `capabilities:`. The **`/review` phase is deliberately thin** — deterministic gates + "dispatch adversarial review *if available*" (§5).

#### 3. `developer-safety` (universal control layer)

The `kill-switch`, `steer`, `commit-on-stop` hooks + `commit-no-coauthor` / `worktrees-never-auto` conventions. `standalone: true`; `enhances: [developer-workflows]`. **AG caveat (carried):** AG plugin hooks are observe/side-effect-only, so `kill-switch`/`steer` are Claude-only-effective; `commit-on-stop` works on both — documented, not worked around.

#### 4. `code-review` (adversarial review, standalone)

The `adversarial-reviewer` + `adversarial-reviewer-cross` agents, the `evidence-tracker` hook (grounds review in what was actually read), `cross-review.sh` (cross-model via the Gemini CLI), and a standalone **`/code-review <diff|PR>`** command. `standalone: true`; `enhances: [{group: developer-workflows, capability: review}]`. Test-sanctity conventions (`tests-are-sacred`, `test-outcome-not-structure`, `verification-executable-first`) are **referenced, not owned** — they live in the future Testing bundle.

#### 5. The auto-enable runtime mechanism + capability discovery

`enhances:` is declarative; enabling happens two ways, by case:
- **Emergent (safety → workflows).** Once `developer-safety` is installed, its hooks are session-global and fire regardless — no detection needed. The `enhances:` entry just *advertises* the relationship for discovery.
- **Conditional dispatch (code-review → workflows' `/review`).** The `/review` phase spec instructs: *"if the `code-review` capability is available, dispatch `adversarial-reviewer` (+ `-cross`); else run deterministic gates only and note the lighter review."* The availability check is a **deterministic probe** (not agent-judgment) — reproducible, graceful-skip when absent (mirrors the memory-seam no-op discipline).

**Generalize the probe into an agentm host feature (the right home).** Rather than each plugin hand-rolling its own install-dir probe, this is better served as a **generalized capability-discovery API in agentm** — the plugin *host* deterministically aggregates installed plugins' declared `capabilities:` and answers *"is capability X available?"* for any plugin at action time. That makes `enhances:` composition first-class and splits cleanly: **`enhances:` is the declarative half** (crickets `group.yaml` schema, this design) and **host capability-discovery is the runtime half** (an agentm-kernel feature — agentm *is* the plugin host). **Cross-repo:** track the discovery API on the **agentm V5-kernel** roadmap (`ROADMAP-AgentMemoryV5.md`); until it lands, plugins fall back to a small **local deterministic probe** (a shared crickets helper) so this suite ships without blocking on the host feature. Either way the runtime contract is the same — a deterministic yes/no with graceful-skip.

#### 6. The `developer`-seed split + sibling re-point + parallel-run

Split the `#40` `developer` seed: control hooks (`commit-on-stop`/`kill-switch`/`steer`) → `developer-safety`; `evaluator` agent → `developer-workflows`; **retire the `developer` group**. Re-point `src/wiki` + `src/github-ci` `requires: [developer]` → `[developer-workflows]`. Regenerate `dist/`; `generate.py check` clean. **Parallel-run discipline:** old agentm copies stay until the plugins are dogfood-proven on agentm/crickets/sherwood (loads via `claude plugin list` / `agy plugin validate`; ≥1 hook fires + a phase command resolves + a standalone `/code-review` runs); the ⑤ V5 slim deletes the agentm duplicates only after proof. **DoD per plugin:** generated byte-deterministically, drift-free, dogfooded on the 3 repos, composition verified (each alone + all combinations).

## Alternatives Considered

- **One mega `engineering-process` plugin (no split).** Rejected — `developer-safety` (any session) and `code-review` (any diff) each have a real standalone use case; fusing them denies those uses. Discipline: split only where a standalone use exists (we would *not* split `explorer`/`evaluator` out — they only make sense inside the loop).
- **Hard `requires:` between the three.** Rejected — they're independently useful; a hard dep forces all-or-nothing. Soft `enhances:` preserves standalone use *and* keeps the enhancee open to new enhancers.
- **"Developer (base)" as a conventions+hooks blob** (the pre-reframe catalog #1 framing). Rejected — it undersells the phase workflow; the *workflow* is the base, not a bag of conventions. `developer-workflows` names it precisely and matches the operator's original "engineering-process" intent.
- **`enhances:` on the enhancee (`enhanced_by:`)** instead of the enhancer. Rejected — declaring on the enhancer keeps the enhancee's manifest closed-to-edit as new enhancers appear (extensible).
- **Agent-judgment availability detection** (the agent just "sees" `@adversarial-reviewer`). Rejected — non-reproducible; a deterministic probe (→ host capability-discovery) gives identical behavior every run.
- **Keep code-review fused into the `/review` phase.** Rejected — review is a first-class, often-standalone concern (the `/code-review ultra` precedent); splitting lets it serve review outside any `/work` cycle.

## Dependencies

- **#40 generator** (`src_model.py` / `generate.py` / emitters / `lint_src.py` / `bootstrap.sh`) — extended for `enhances:` + `capabilities:`, not replaced.
- **The agentm extraction source** — the phases / commands / sub-agents / hooks / `.harness/` scaffold / conventions live in agentm today; copied out **parallel-run** (originals removed by the ⑤ V5 slim, not here).
- **Gemini CLI** — `code-review`'s cross-model `adversarial-reviewer-cross` shells out via `cross-review.sh`; graceful-skip (or operator-editable to `claude`) when absent.
- **Soft / future — the agentm capability-discovery API** (§5): the runtime half of `enhances:`. Tracked on the agentm V5-kernel roadmap; **local deterministic-probe fallback** until it lands, so this suite is not blocked on it.
- **Parent / siblings** — the [memory-os-architecture HLD](https://github.com/alexherrero/agentm/wiki/memory-os-architecture) (parent) + ROADMAP-MASTER bucket ④; the existing `wiki` / `github-ci` groups (their `requires:` re-points here).

## Migrations

- **Seed split.** The `#40` `developer` group splits: control hooks → `developer-safety`, `evaluator` → `developer-workflows`; **`developer` is retired**. `src/wiki` + `src/github-ci` `requires: [developer]` → `[developer-workflows]`. `dist/` regenerated; `generate.py check` clean.
- **Schema additive.** `group.yaml` gains optional `enhances:` + `capabilities:` — existing groups (`pii`, the re-pointed ones) are unaffected if they omit them. No breaking change to the manifest schema.
- **No data / user-state migration.** The `.harness/` scaffold is created at runtime by `/setup`; there's no pre-existing state to migrate. Config is the per-plugin `settings.default.json` seed (create-if-absent) from the HLD.
- **Parallel-run, not a flag day.** agentm keeps its baked-in phases/agents/hooks through this bucket; the duplicate-removal (slim) is ⑤ V5's job, gated on dogfood-proof — so a half-built suite never strands the operator.

## Technical Debt & Risks

- **Local-probe fallback is accepted debt** until the agentm capability-discovery API ships — probe logic duplicated across plugins; **re-audit when the host feature lands** (then plugins query the host and the local probes retire).
- **Parallel-run duplication** — agentm + the plugins both carry the workflow until the ⑤ V5 slim; a window of two sources of truth. Mitigated: the plugins are the canonical go-forward + the slim is a tracked next bucket.
- **`enhances:` is a crickets-only concept** (no host primitive). **Re-audit trigger:** if a host ships native soft-composition (mirrors ADR 0013's hard-dep trigger), reconsider.
- **`capabilities:` drift** — a capability named in an `enhances` entry must match the target's declared `capabilities:`; if a plugin's commands change but its declared list doesn't, `enhances` targeting goes stale. Mitigated by `lint_src` validating the cross-reference at build.
- **AG observe-only hooks** (carried) — `kill-switch`/`steer` are ineffective on Antigravity; shipped + documented, not worked around. Re-audit if AG ships hook-veto.
- **Cross-model review depends on the Gemini CLI** + operator gemini auth; graceful-skip to same-model review, or operator edits `cross-review.sh` to `claude`.
- **Scope** — three plugins + a schema change is a large ④ wave; mitigated by the translate part-split (incremental ship) + `developer-workflows` being usable before safety/code-review land.

## Quality Attributes

*Per-attribute analysis. Each sub-section below is **mandatory**: either describe the concern or explicitly mark N/A with a one-sentence justification.*

### Security

Extracted hooks/agents run with operator privileges (unchanged from agentm today). `code-review`'s `adversarial-reviewer-cross` shells out to the Gemini CLI (subprocess; operator-invoked, bounded). The deterministic capability probe is read-only (checks install dirs). No new network surface beyond the existing Gemini shell-out. Single-operator, locally installed. PII guardrails apply to every extracted file (crickets is public).

### Reliability

The defining property: **graceful composition.** Each plugin works alone; `enhances:` is purely additive; the conditional dispatch (code-review → `/review`) graceful-skips when absent; the local-probe fallback degrades to a clean no-op. **Parallel-run** means a half-built suite never strands the operator (agentm's baked-in copies remain until the ⑤ slim). Generator determinism + `generate.py check` drift gate + per-plugin dogfood DoD. A misbehaving plugin is a `/bugfix`; uninstall removes it.

### Data Integrity

`commit-on-stop` side-ref snapshots remain the working-tree recovery net (safety plugin). The only state these plugins write is the `.harness/` scaffold (workflow plugin, via `/setup`) — **no vault writes** (memory is agentm's domain). `settings.default.json` seeding is create-if-absent (never clobbers operator edits). The extraction preserves git history (copy-then-delete-after-proof; the delete is the ⑤ slim).

### Privacy

crickets is **public** → extracted files carry **zero PII** (scrubber + pre-push + CI gitleaks). The plugins operate on the operator's repos locally; no telemetry. **Flag:** `adversarial-reviewer-cross` sends the diff to the Gemini CLI (→ Google) when run — existing agentm behavior, **operator-opt-in per invocation**, and editable to `claude` in `cross-review.sh`.

### Scalability

N/A: single-operator toolkit; the generator runs sub-second over `src/`; this adds three plugin groups + one schema field.

### Latency

Real but narrow: `developer-safety` hooks sit on the session/tool-call path (kill-switch/steer PreToolUse, commit-on-stop Stop) and must stay fast + pure-local; `evidence-tracker` (code-review) is PreToolUse, sub-100ms. The capability probe runs once per `/review` dispatch (local file check). **No network on any hot path** — cross-review is explicit/offline.

### Abuse

N/A: single-operator local tooling; no untrusted-input surface.

### Accessibility

N/A: text/CLI tooling, terminal output only.

### Testability

Central — it *is* the DoD. **Composition matrix:** each plugin alone + every combination (workflows-only · +safety · +code-review · all three) — assert graceful-skip *and* enhancement. The `enhances:`/`capabilities:` **lint rules are unit-testable** (target exists · no self-enhance · `enhances ∩ requires = ∅` · capability declared). `generate.py` byte-determinism + drift gate. The deterministic probe is testable (present/absent → correct yes/no). Dogfood on agentm/crickets/sherwood (loads · hook fires · phase command resolves · standalone `/code-review` runs). Falsifiable, not LLM-judge.

### Internationalization & Localization

N/A: operator-personal, English-only toolkit.

### Compliance

N/A: no regulatory scope; public-repo discipline is covered under Privacy.
