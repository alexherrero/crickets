# Crickets Composition Design

## Objective

Crickets is a box of stateless tools, yet the work it does holds together — `/release` knows the plan `/work` built, `code-review` sharpens what `/review` runs. This design specifies the mechanism behind that coherence: **composition** — how tools combine by capability-name, resolved at load, with every dependence pointing one way.

## Overview

Three kinds of composition meet here: capability with capability, capability with an agentm opinion, and the whole toolbox with the person (agentm). All three are **by name** and **one-way** — a unit names the capability it wants, the runtime wires it at load, and dependence points up the stack. A tool reaches another tool's surface, or an opinion, or the substrate, by asking the resolver for it; each plugin's code stays in its own plugin (the soft-composition rule, established for the developer suite and now held here).

The target shape is **thirteen plugin-capabilities**, and **thirteen plugins ship today** (the sets differ). The target renames `developer-workflows` → `development-lifecycle` (the spine, now owning the feature's whole arc — `launch` / `deprecate` / `retire` included), folds `testing-conventions` + `releasing-conventions` → `conventions` and `status-line-meter` → `token-audit`, and adds the two new `research` / `diagnostics`. ## Design

### Capabilities compose with each other

A capability names two kinds of relationship to another, by capability name:

- **`requires:`** — a hard dependency: it will not run without the target (e.g. `github-projects` requires `developer-workflows`).
- **`enhances:`** — a soft dependency: it works alone, and does more when the target is present (e.g. `code-review` enhances `developer-workflows`'s `/review`).

At load time agentm's resolver (`capability_resolver.py`) matches names to installed capabilities, and the crickets bridge (`find_capability.py`) reaches it. These links arise where a primitive is shared across capabilities, or one capability builds on another's surface.

**How `enhances:` resolves.** The relationship is asymmetric and order-free: the enhancer names the target, the target knows nothing of its enhancers, and the resolver matches at load regardless of install order — installing the enhancer before or after the enhancee gives the same result. A soft match that finds nothing resolves to a **quiet absence**, so any subset of crickets installs and runs. The declared edges also drive discovery surfaces: `suggest_enhancers.py` (governed here) lets `bootstrap` suggest installing an enhancer ("you have `development-lifecycle`; install `code-review` for the full experience"), and the marketplace render carries the same "works better with …" links.

**The honesty lints** (in the generator's `lint_src.py`) keep the graph sound: the named target must exist, nothing enhances itself, `enhances:` and `requires:` cannot name the same capability, and a named target must be declared in some plugin's `capabilities:`. A capability declares the names it provides in `capabilities:`; those are what others may target. All thirteen shipping plugins declare `capabilities:` today.

**Version matching** — `capability_version_match.py` (a single-range semver check: `>=`, `>`, `<=`, `<`, `==`, `!=`, `~=`, degrading to a non-match on malformed input rather than raising) exists in agentm. **`[PENDING-IMPL]`** — crickets does not yet wire it in: the `Enhance` dataclass (`scripts/src_model.py`) carries no version field, so there is no schema slot to name a version range on an `enhances:` relationship today, and nothing in crickets calls the matcher. The open policy calls (enforce at load vs. call · warn vs. error) are in Risks, alongside the schema/call-site gap.

### Capabilities compose with opinions

The same `requires:` / `enhances:` mechanism reaches *up* to agentm's [opinions](https://github.com/alexherrero/agentm/wiki/agentm-opinions-and-gates). A capability — or a single primitive — can **depend on** an opinion (its gate needs *what "done" looks like* to run at all) or be **enhanced by** one (it produces better output when *what "good" looks like* is available, and still runs without it). When the opinion is present it shapes the work; when absent, the capability degrades gracefully.

The opinion surfaces (done / good / efficient / how-we-engineer) and the request-by-name path a tool would query live in the [agentm Opinions design](https://github.com/alexherrero/agentm/wiki/agentm-opinions-and-gates) — **the resolver itself is built** (`opinion_resolver.py` + the nine `opinions/*.md` stubs, AG Wave B), but only **2 of the 9 originally-scoped consumer bindings across the crickets portfolio are actually wired** (PLAN-wave-d-opinion-wiring, 2026-07-06):

- **Wired:** [diagnostics](crickets-diagnostics.md)'s `writer.py` requests `how-we-engineer` and folds the resolved body into every written incident. [development-lifecycle](crickets-development-lifecycle.md)'s routed-dispatch layer discovers `tokens/scripts` via `opinion_resolve('efficient')['implements']` instead of a hardcoded path (the concrete tier/model/effort data still comes from `token-audit`'s own routing table — the opinion's `implements:` artifact, not a duplicate).
- **Interpolated (markdown-prose consumer grammar, PLAN-opinion-consumer-grammar task 2, 2026-07-07):** `design`'s `good` binding — `src/design/commands/design.md`'s Step 6 (Review pass) now declares `opinions: [good]` and carries an `<!-- opinion:good --> … <!-- /opinion:good -->` marker that `generate.py` splices at build time from a committed snapshot (`scripts/opinion-snapshots/good.md`), baked into both `dist/claude-code/` and `dist/antigravity/`. This is the grammar's first real cross-plugin consumer — see the [build-system design](crickets-build-system.md)'s Amendment log for the mechanism.
- **Declared self-provider, staying hardwired by design (PLAN-opinion-consumer-grammar task 3, 2026-07-07):** `code-review`'s `good` binding — `adversarial-reviewer.md` now declares `opinions: [good]` frontmatter. Unlike the interpolated case above, this stays hardwired with **no** splice: code-review IS `good`'s `implements:` artifact (a self-provider, not a consumer reusing someone else's text), so the grammar calls for a drift check only (`scripts/check-opinion-self-provider-drift.py`, flagging agentm's `opinions/good.md` stub if it drifts from this file's own prose — never the reverse). The distinction from the still-fully-undeclared row below: this binding is now explicit and monitored, not silently implicit — but "hardwired" is still the correct terminal state, not a waypoint to interpolation.
- **Still hardwired and undeclared, staying that way until a real design gap closes:** `research` (`how-we-engineer`), `maintenance` (`done`), `wiki` (`good`), `design`'s own **`how-we-engineer`** binding (its `good` binding moved to Interpolated above — the same file now carries one baked opinion and one still-hardwired), and development-lifecycle's own direct `done`/`how-we-engineer` bindings. **Why:** each of these capabilities' opinion text lives in a static markdown skill/command/agent file — the file *is* the prompt, loaded verbatim by the host, with no runtime render step to inject a resolved opinion body into, and (unlike `code-review`) none has yet been classified self-provider vs. cross-plugin or given `opinions:` frontmatter. The **markdown-prose consumer grammar** that used to block all of them is now locked and proven on one binding of each class (cross-plugin: `design`→`good`; self-provider: `code-review`→`good`, both above); moving the rest is mechanical fan-out per binding, not further design work.
- **Dropped, not deferred:** `privacy`'s `privacy-review` skill (the skill itself is `[PENDING-IMPL]` — nothing exists yet to wire; its opinion binding belongs in whichever plan builds it) and `token-audit` (it's `efficient`'s one-way measurement *provider*, not a consumer with hardwired guidance text — the original staging research misread the direction of that arrow).

**The follow-up shape, now proven rather than merely named:** build-time interpolation via `generate.py`'s emit pipeline (`scripts/src_model.py`'s `interpolate_opinions()` / `render_primitive_text()`), reading a primitive's `opinions:` frontmatter key and splicing a committed snapshot's body into an HTML-comment marker pair in the source, resolved at build time — mirroring how `style_resolver.py` already composes wiki's voice overlay at author time, just shifted to build-time since these files have no author-time render step. Locked and shipped on `design`→`good` (PLAN-opinion-consumer-grammar task 2, 2026-07-07). The self-provider half of the grammar (declare `opinions:`, no splice, drift-check only) is separately proven on `code-review`→`good` (task 3, same date). Fanning either shape out to the remaining undeclared bindings (`research`, `maintenance`, `wiki`, `design`'s `how-we-engineer`, development-lifecycle's direct bindings) is mechanical repetition, not new design work — task 4 of the same plan grounds whether `maintenance`/`wiki` even have a real prose site to classify before that fan-out starts.

### The toolbox composes onto agentm — the one-way arrow

**crickets depends on agentm; agentm never depends on crickets.** That arrow is the central rule of the whole split. A tool draws on the substrate by *asking*: the authoritative resolver lives in agentm (`capability_resolver.py`); crickets reaches it through a thin bridge (`find_capability.py`) that discovers it by path-fallback and skips gracefully when agentm is absent. The graceful skip runs both ways — remove the tools and agentm still remembers; remove agentm and the tools still run, just without memory-backed continuity. A bare agentm is whole on its own.

Three decisions drew the arrow: the repo split made them siblings sharing only a byte-identical `lib/install/`; the V5 unbundling deleted the phase loop from agentm and re-homed it in crickets, removing even the hard-require; and the routing-plane de-vaulting kept agentm's substrate speaking in backend-agnostic terms, so a tool can ask without assuming vault paths (V5-6, in the `memory-storage-seam` living design). *(The first two decisions now live in the [agentm HLD](https://github.com/alexherrero/agentm/wiki/agentm-hld) amendment log. Transitional wrinkle: until the orchestration-split slim lands, agentm still carries baked-in copies of some dev-loop scripts from before the unbundling — a cleanup-status detail.)*

### Role-retirement (the transitional shape)

There is **no role tier**. What used to be called a "role" is a **persona** (an agentm concept — see the [agentm Personas design](https://github.com/alexherrero/agentm/wiki/agentm-personas)) wielding a package of crickets tools. The agent-definitions the phase loop ships today (`worker.md`, `tech-lead.md`, …) are transitional: their *stance* migrates up into agentm personas, and crickets keeps the *tools* plus the package — correlated tools named after the persona that wields them. The read-only helpers (`explorer`, `evaluator`) stay tools, by the cross-capability-judgment discriminator. This resolves the persona-vs-role question the launched developer-suite design (since subsumed into this design) deferred; `Coordinator-Roles.md` and the agent-defs reconcile to the persona-wields-tools shape when role-retirement lands.

### Renames keep both old and new names resolving

A v6.0 capability rename — `developer-workflows` → `development-lifecycle`, `github-ci` → `maintenance`, `design-docs` → `design`, `wiki-maintenance` → `wiki`, `token-audit` → `tokens`, `pii` → `privacy`, plus the `testing`+`releasing` → `conventions` merge — does **not** break the existing `requires:` / `enhances:` edges that named the old capability. There is **no separate alias map**: the renamed (or merged) plugin's `group.yaml` simply **declares both the old and the new capability name** in its `capabilities:` list, and the resolver — which matches by declared name — resolves either. The old name is dropped in a later major once no consumer references it (a no-dangling-name check at that point). The seven renames all reference this one mechanism; it is what their Migrations sections mean by "keeps the old names resolving."

## Dependencies

- **agentm's resolver, via the bridge** — composition's by-name lookups resolve through agentm's `capability_resolver.py`, reached by crickets' `find_capability.py` (path-fallback + graceful-skip). The one-way arrow: crickets depends on agentm, never the reverse.
- **The generator's lints** — `lint_src.py` enforces the four honesty lints over every plugin's `requires:` / `enhances:` / `capabilities:`; the [crickets build-system design](crickets-build-system.md) owns the generator.
- **Version matching** — `capability_version_match.py` (agentm) exists, but crickets doesn't yet wire it in. **`[PENDING-IMPL]`** — the `Enhance` dataclass has no version field, so no relationship can name a version range on its target today.
- **Sibling designs** — [agentm Opinions](https://github.com/alexherrero/agentm/wiki/agentm-opinions-and-gates) (the opinion surfaces + the request-by-name path), [agentm Personas](https://github.com/alexherrero/agentm/wiki/agentm-personas) (the persona tier that wields tool packages), [crickets build-system](crickets-build-system.md) (where the lints + `capabilities:` are generated).

## Risks & open questions

- **Version-matching policy is unsettled** — the matcher exists, but whether a range is enforced at load or at call, and whether a mismatch warns or errors, is the per-capability sub-designs' to settle. More basically, it's **`[PENDING-IMPL]`** in crickets today: `Enhance` has no version field and nothing calls the matcher, so there's no schema slot for a range to enforce yet — the policy question is moot until the wiring lands.
- **The opinion request-by-name path is built, but only 2 of 9 consumer bindings are wired onto it** — the resolver itself shipped in AG Wave B; the remaining 6 hardwired bindings are blocked on an unlocked markdown-prose consumer grammar (see "Capabilities compose with opinions" above for the full breakdown), not a Phase-3/4 timeline that will resolve itself as other work lands.
- **`research` and `diagnostics` are newly designed, not yet built** — their precise depends/enhances are carried from their authored sub-designs; the edges land as they ship.
- **Role-retirement is mid-flight** — `Coordinator-Roles.md` + the transitional agent-defs (`worker.md`, `tech-lead.md`, …) reconcile to the persona-wields-tools shape when role-retirement lands.
- **A transitional baked-in-scripts wrinkle** — until the orchestration-split slim lands, agentm still carries copies of some pre-unbundling dev-loop scripts.
- **Re-audit triggers:** re-derive the relationship map from `dist/default-set.json` as the set grows; confirm the three new capabilities' edges when their sub-designs land; reconcile `Coordinator-Roles.md` at role-retirement; convert the mermaid map to a vector diagram at the parent's diagram pass; repoint the ADR references as agentm 0006/0011/0016 fold into living designs.
